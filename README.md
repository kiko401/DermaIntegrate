# DermaIntegrate: 皮肤病多源数据集成与智能辅助诊断系统

**许可协议**：MIT License 
**运行环境**：Python 3.11 / Java 17+ (Spring Boot 3.x+) / Node 18+ 
**系统架构**：基于 API 契约的模块解耦，推理服务与应用服务进程级隔离与独立部署

---

## 摘要

本项目面向皮肤病专科门诊场景，解决临床业务中多源异构数据孤岛、初诊数据缺失与医疗 AI 缺乏可解释性的问题，构建数据集成与多模态辅助诊断系统。

在**数据集成层面**，针对 HIS、PACS、LIS 等系统数据异构与标识不一的问题，系统采用主索引映射与 FHIR R4 标准协议，实现跨系统异构数据的归一化接入与逻辑聚合，构建患者全生命周期临床视图，且不侵入源系统。

在**AI 辅助层面**，针对单模态图像推断局限与端到端模型缺乏推断依据的问题，提出基于多 Agent 协作与检索增强的多模态可解释机制。系统接收皮肤镜图像、病历文本与病理分子数据等多模态输入，由独立的图像 Agent（病灶分割与特征提取）、病历 Agent（结构化解析）、病理与分子 Agent（基于指南的精准分期规则引擎）按需处理，最终由整合 Agent 融合多源特征与 RAG 两阶段检索结果，生成结构化风险提示与检查建议。系统实现了对数据缺失场景的良好兼容，在输入不完整时能给出定向补充检查建议并降级报告状态，诊断裁量权交由临床医生。全管线实施全中文强制防线与强引用约束，有效防范术语漂移与知识幻觉。

在**工程实现层面**，系统针对医学影像前端解析瓶颈、DICOM 标签不规范风险与长时 AI 推理引发的资源占用问题，分别实现了基于成像特性的防御性后端转码与前端轻量渲染机制，以及基于 SSE 的异步响应代理与断连感知计算资源回收机制。同时，在智能推理域低资源云服务器环境下，通过容器内存限制、ONNX 推理替换与国内镜像源注入实现稳定部署，保障临床环境下的系统可用性。

## 1. 核心问题与方法映射

针对皮肤病临床业务中的数据流转与 AI 应用问题，本系统的应对策略与工程机制如下：

| 临床工程问题                                                 | 本系统应对策略与工程机制                                     |
| :----------------------------------------------------------- | :----------------------------------------------------------- |
| **多源异构数据孤岛**：HIS、LIS、PACS等系统数据异构，患者标识不一，难以形成完整临床视图。 | **主索引映射与 FHIR 标准归一化**：基于主索引映射跨系统患者标识，结合 FHIR R4 标准协议解析资源，实现异构数据的逻辑聚合与患者全生命周期视图构建，不侵入源系统。 |
| **初诊数据缺失与单模态局限**：初诊时常缺乏病理金标准，单一图像模态无法支撑完整分期；强制要求全量数据导致系统不可用。 | **多模态缺失兼容与定向建议**：系统以任务驱动多模态管线，各 Agent 按需触发，跳过缺失模态。整合 Agent 针对缺失项动态生成定向补充检查建议，报告状态降级，不因数据不全而阻断流程。 |
| **医疗 AI 缺乏可解释性与幻觉风险**：端到端模型直接生成诊断结论缺乏推理依据；LLM 易产生术语英文漂移与无源头引用。 | **多 Agent 证据链与全中文防幻觉防线**：图像 Agent 仅输出客观形态学特征严禁下诊断，病理 Agent 基于指南输出规则分层，整合 Agent 结合 RAG 强引用生成报告。全管线实施全中文强制映射与自重试机制，系统拒绝在数据不足时给出终局诊断。 |
| **DICOM 影像前端解析与标签风险**：PACS 下发的影像强依赖重型解析库实时渲染导致卡顿；且公开数据集常存在标签不规范导致严重偏色。 | **防御性影像转码与轻量渲染**：基于皮肤镜无需调窗的特性，后端离线解析 DICOM 并执行防御性色彩空间还原转码为 PNG，结合前端轻量滑动对比组件，实现影像数据的透明集成调阅。 |
| **长时 AI 推理引发的资源占用**：AI 推理耗时导致服务端计算资源长期占用，并发能力衰减，且异常断连易导致计算浪费与 OOM 崩溃。 | **异步响应代理与低资源部署**：基于 SSE 异步契约解耦业务与推理，建立断连感知逻辑，前端断连即终止推理计算；智能推理域云端采用低资源部署，通过 ONNX 推理替换与容器内存限制保障系统稳定性。 |

## 2. 系统架构设计

系统遵循“推理计算与业务流转解耦”原则，划分两大进程级隔离服务域：**智能推理域**负责多模态 AI 推理、特征提取与影像预处理；**应用平台域**负责数据集成归一、标准协议接入与前端交互。

```markdown
③ 表现层（应用平台域 - React 18 + AntD + ECharts）
  ├─ 医生工作站 (患者全生命周期临床视图聚合展示)
  ├─ 影像滑动对比组件 (原图/标注图叠加渲染)
  └─ AI 辅助诊断流式对话框 (展示多模态证据链，动态渲染缺失降级状态)

② 服务与数据层（跨域协同）
  ├─ 智能推理域 Multi-Agent 协作管线：
  │   ├─ 图像 Agent：病灶分割提取 | 形态学特征提取与全中文约束
  │   ├─ 病历 Agent：专科 Schema 映射 | 自由文本解析与防漂移自重试
  │   ├─ 病理与分子 Agent：基于 AJCC/NCCN 的精准分期规则引擎与缺失兜底
  │   └─ 整合 Agent：动态 Prompt 构建 | RAG 两阶段检索 | 结构化报告生成与 JSON 容错修复
  ├─ 应用平台域：患者临床视图聚合 | 主索引映射 | FHIR R4 协议接入 | AI 诊断异步代理
  ├─ 推理域数据核心：独立数据库 (Task 任务映射、多模态上下文存储)
  └─ 应用域数据核心：MySQL 8 (关系型业务数据存储)

① 接入与预处理层
  ├─ DICOM 离线摄取管线、防御性色彩空间还原、像素转码与 UID 绑定 (推理域执行)
  └─ FHIR Mock Server (仿真 HIS/EMR/LIS，应用域接入)

⓪ 基础设施层
  ├─ 智能推理域 Docker Compose 编排、容器内存限制与宿主机 Swap 配置
  └─ Nginx 网关路由与 SSE 长连接超时配置
```

## 3. 关键工程实现与核心机制

### 3.1 多 Agent 协作与多模态缺失兼容机制

为解决初诊数据缺失与单模态推断局限的问题，系统将图像单管线重构为以 `task_id` 驱动的多 Agent 协作管线。系统接收多模态数据，各模态由独立 Agent 按需处理，最终由整合 Agent 融合输出，全程兼容数据缺失：

1. **任务驱动与条件分支调度**：API 入口接收任意组合的多模态数据并生成唯一 `task_id`落库。SSE 推理流根据数据存在性按需触发子 Agent，无图像则跳过视觉定位，无病理则跳过分期引擎。
2. **图像 Agent（视觉定位与形态提取）**：
   - **病灶分割与特征提取**：采用 ONNX Runtime 进行 CPU 推理。针对模型输出的离散假阳性噪声，工程层强制实施最大连通域提取，剔除微小噪声，生成精准的病灶掩码。最终输出病灶区域占比、纯中文位置及热力图叠加。**降级机制**：若部署环境中无权重文件，系统自动降级至高斯模糊掩膜占位符，保障全管线不中断。
   - **形态学结构化约束与全中文防线**：视觉语言模块严格降级为“特征提取器”，通过 Prompt 与后置正则过滤双重约束，剥离诊断性结论并拦截英文术语漂移，确保视觉模块的客观性。
3. **病历 Agent（结构化映射与防漂移机制）**：
   - **专科 Schema 映射**：结构化通道严格对齐核心临床字段，内置强制中文映射字典与防御性布尔清洗，从源头杜绝英文漂移。
   - **LLM 防漂移自重试机制**：针对 LLM 常见的语义漂移，实施“带反馈的局部自重试”。首次提取后由规则引擎校验，若不通过则将错误原因作为反馈再次请求 LLM 修正，达到最大重试次数仍失败时返回空 Schema 兜底，确保管线不阻塞。
4. **病理与分子 Agent（规则引擎与跨模态协同）**：
   - **精准分期与核心判空**：纯规则引擎实施**判空阻断原则**：若核心浸润深度指标为空，分期逻辑阻断；若次要状态未知，采用**临床保守原则**按高危倾向处理，宁预警不漏诊。
   - **跨模态依赖逻辑**：规则引擎支持跨 Agent 数据引用，如依据病历 Agent 提取的部位特征，触发特定亚型附加建议，实现跨模态规则协同。
5. **整合 Agent（动态判定与全中文终极防线）**：
   - **动态缺失判定**：接收各子 Agent 输出，若某模态缺失，Prompt 明确指示优先考虑完善相关确诊检查。引入 `status: "incomplete" / "complete"` 逻辑，当核心金标准缺失时，系统强制覆写为 `incomplete`。
   - **全中文终极防线与强引用**：Prompt 强制约束严禁夹杂英文术语。结合 RAG 返回的带前缀 ID 文本，要求 LLM 照抄引用，有效抑制知识幻觉。结合强制 JSON 输出与运行时容错修复机制，保障前端契约稳定交付。
6. **RAG 两阶段检索（特征硬匹配与向量精排）**：
   - **强制 ID 标注与预打标**：知识库语料采用前缀式独立编号，每条语料末尾强制附带分类标签。
   - **特征映射与硬匹配筛选 (Stage 1)**：检索前，将 Agent 的中文输出映射为检索标签，基于标签构建硬匹配筛选子集，极大提升专科指南信噪比。
   - **复合查询与向量精排 (Stage 2)**：将多模态特征拼接为自然语言复合查询，在筛选子集内基于向量精排 Top-3，返回结果强制将 ID 拼回文本头部供 LLM 引用。

### 3.2 主索引映射与标准协议的数据集成机制

针对多源异构数据孤岛，系统在应用平台域实现基于映射路由的归集与标准协议适配，不对外部异构库做侵入性写入：

1. **标准协议接入**：通过适配器解析符合 FHIR R4 规范的资源，将非标准化的临床数据映射为标准资源，标准化临床数据入口。
2. **交叉标识提取与多级匹配**：从各源系统数据中提取患者核心身份标识，首选身份证号精确匹配，辅以姓名、手机号及出生日期组合确认，解决跨系统患者标识不一致问题。
3. **主索引映射与只读路由**：构建 EMPI 映射元数据表，将不同源系统的 ID 与内部全局 ID 建立关联。系统仅存储路由映射关系，通过标准 API 或只读视图路由获取详情，在应用层完成多源数据的逻辑拼装。

### 3.3 基于成像特性的防御性影像转码与前端内存管控

PACS 下发的 DICOM 影像在前端解析存在性能瓶颈，且公开数据集存在标签不规范风险，系统实施防御性轻量化集成策略：

- **防御性后端离线转码与色彩空间还原**：皮肤镜影像无需调窗。针对公开数据集常见的标签与底层实际存储不一致导致的偏色风险，管线采用防御性映射策略：对 3 通道图像直接按位深缩放输出，无视可能误导的标签；对单通道灰度图严格按标签处理。同时应用像素归一化映射，确保色彩无失真，转存为 Web 友好的 PNG 格式。
- **摄取接口与 UID 绑定**：推理域提供上传与离线 URL 摄取接口。原始文件落盘后同步调用解析器转码，提取关键元数据在自有数据库建立映射记录，将 UID 与转存路径强绑定。
- **前端渲染与内存生命周期管控**：针对高分辨率影像导致的浏览器内存泄漏风险，前端采用按需加载机制，并在组件卸载或上下文切换时主动释放 Blob 对象资源，防止内存泄漏。

### 3.4 异步响应代理与断连感知计算回收机制

医疗 AI 推理耗时较长，为避免资源浪费与 OOM 风险，系统构建了端到端的异步与资源回收逻辑：

- **应用域响应式代理**：应用域规避同步阻塞调用推理域，通过响应式客户端转发请求，将 SSE 流反向代理给前端，防止线程池耗尽。
- **SSE 流多步级状态契约**：SSE 通道严格定义事件类型契约，支持异步步级进度推进（如 `image_done`, `clinical_done`, `pathology_done`, `result` 等）。前端采用动态渲染窗口，无需等待缺失模态。
- **断连感知与流式步级推理终止逻辑**：当医生关闭页面或网络断开时，应用域感知中断并停止拉取流。推理域将耗时任务解耦至独立线程，主协程通过异步监测连接状态，一旦断连即刻通过事件信号终止推理循环，释放计算资源。

### 3.5 数据存储与智能推理域低资源环境管控

针对多模态数据结构与低资源部署场景，实施规范化的存储建模与轻量级资源管控：

- **推理域数据模型扩展**：为适配多模态接入，数据库表结构扩展为 `Task` 语义，新增多模态上下文字段，图片路径允许为空以兼容无图场景。
- **智能推理域低资源部署策略**：
  1. **ONNX 替换 PyTorch 推理**：使用 ONNX Runtime 进行 CPU 推理，大幅降低内存占用与依赖体积。
  2. **容器与系统级限制**：宿主机配置 Swap 防止系统 OOM；通过编排文件强制限制基础组件内存上限，给推理服务预留计算空间。
- **云端环境部署约束**：
  1. **基础版本约束**：因依赖兼容性要求，基础镜像需采用 `python:3.11-slim`。
  2. **向量化模型源配置**：国内部署需配置 `HF_ENDPOINT` 环境变量确保模型正常拉取。
  3. **无头模式依赖**：容器环境需使用 Headless 版本 OpenCV。
  4. **核心依赖版本锁定**：需严格锁定 NumPy 等底层库的大版本号，以兼容 ONNX Runtime 与 OpenCV 的编译约束。

## 4. 协作范式与项目拓扑

本项目由两位协作方共同完成，采用 API 契约驱动开发模式，服务域严格隔离。智能推理域聚焦于 AI 工程与数据治理，应用平台域聚焦于全栈高可用与医疗标准落地。

### 4.1 角色矩阵

| 协作方     | 核心职责域                                            | 工程侧重                                                     |
| :--------- | :---------------------------------------------------- | :----------------------------------------------------------- |
| **算法端** | 智能推理域（多 Agent 子系统、影像预处理、RAG 知识库） | 多模态 Agent 编排、全中文防漂移防线、专科规则引擎、DICOM 转码管线与两阶段检索 |
| **工程端** | 应用平台域（业务服务、前端交互、数据访问、协议接入）  | 异构数据集成归一、异步代理高可用保障、临床交互与内存管控     |

## 5. 部署与运行

### 5.1 环境配置

智能推理域采用 Docker Compose 编排，并针对低资源云服务器进行了深度适配。前提条件：推理域面向 CPU 环境部署，无需宿主机 GPU。建议宿主机配置 4GB Swap 防止 OOM。

```bash
git clone https://github.com/yourname/DermaIntegrate.git
cd DermaIntegrate
```

### 5.2 参数配置

启动服务前需配置必要参数，**请严格区分智能推理域与应用平台域的数据库与网络边界**。

**1. 智能推理域配置**：在 `backend-ai/.env` 中配置大模型接口、数据库与降级开关（参考 `.env.example`）：
```env
# --- 推理域数据库配置 (独立于应用域数据库) ---
DATABASE_URL=mysql+aiomysql://root:root_password@localhost:3307/derma_ai
# Docker部署时请切换至容器内部网络连接

# --- 视觉语言模型配置 (用于形态学特征提取，严禁下诊断) ---
VLM_API_KEY=your_deepseek_api_key
VLM_BASE_URL=https://api.deepseek.com
VLM_MODEL=deepseek-vl2

# --- 整合/病历提取大模型配置 ---
# 采用非思考模式以保证 JSON 输出格式稳定
INTEGRATION_API_KEY=your_deepseek_api_key
INTEGRATION_BASE_URL=https://api.deepseek.com
INTEGRATION_MODEL=deepseek-v4-flash

# --- 降级开关配置 ---
# 视觉模块处于占位降级状态时设为 true，待真实权重替换后改为 false
USE_MOCK_VLM=true
USE_MOCK_INTEGRATION=false

# --- RAG 向量数据库配置 ---
QDRANT_HOST=localhost
QDRANT_PORT=6333
QDRANT_COLLECTION=derma_knowledge

# --- Docker 环境标识 (云端部署务必设为 true) ---
DOCKER_ENV=false

# --- HuggingFace 镜像源 (国内云服务器部署必填) ---
HF_ENDPOINT=https://hf-mirror.com
```

**2. 应用平台域配置**：在 `backend-app/src/main/resources/application.yml` 中配置：
```yaml
spring:
  datasource:
    # 应用域独立数据库，默认 3306 端口，与推理域物理隔离
    url: jdbc:mysql://localhost:3306/derma_integrate?useSSL=false&characterEncoding=utf
    username: root
    password: root_password

# 智能推理域路由地址
ai:
  pipeline:
    base-url: http://localhost:8000
```

### 5.3 运行模式

**全链路联合启动**：
构建完整的数据集成与辅助诊断验证环境。若部署在单台服务器，请确保满足智能推理域资源要求。
```bash
docker-compose up -d
```
启动后访问 `http://localhost/` 进入医生工作站，内置 Mock Server 支持从数据接入到辅助诊断的完整流程验证。

**智能推理域独立调试**：
适用于算法迭代与推理逻辑验证，无需启动 Java 后端与前端。
```bash
# 启动依赖组件
docker-compose up -d mysql qdrant

cd backend-ai
python init_rag.py  # 初始化 RAG 知识库
uvicorn api.main:app --reload --port 8000
```

**智能推理域云端极限部署**：
适用于推理域独立上云，应用平台域通过公网跨域调用。
```bash
# 1. 宿主机配置 4GB Swap (防 OOM)
sudo fallocate -l 4G /swapfile && sudo chmod 600 /swapfile && sudo mkswap /swapfile && sudo swapon /swapfile

# 2. 修改 .env 配置：启用 Docker 部署专用的 DATABASE_URL、设置 DOCKER_ENV=true 及 HF_ENDPOINT

# 3. 启动容器 (在 infra 目录下)
cd infra && sudo docker-compose up -d

# 4. 初始化 RAG 知识库
sudo docker exec -it derma-fastapi python init_rag.py

# 5. 配置 Nginx 反向代理：需开启 proxy_buffering off 与 proxy_read_timeout 300s 保障 SSE 流
```

**应用平台域独立调试**：
适用于业务逻辑与前端交互调试，AI 请求可路由至云端或本地推理服务。
```bash
# 启动依赖组件
docker-compose up -d mysql redis

# 启动后端服务
cd backend-app && mvn spring-boot:run

# 启动前端界面
cd frontend && npm install && npm run dev
```

## 6. 架构约束与设计原则

为保障多源数据集成的准确性、多模态输入的兼容性及 AI 辅助的可解释性，本项目实施以下架构约束：

1. **多模态缺失兼容与定向指导原则**：系统对图像、病历、病理数据的缺失具备容忍度。当核心确诊数据缺失时，系统不阻断流程，必须给出明确的定向补充检查建议，系统拒绝在数据不足时给出终局诊断，并标记降级状态。
2. **可解释证据链与防幻觉防线**：系统仅提供视觉定位、形态学特征与规则分层作为辅助，不直接输出诊断结论。图像模块受强制格式约束并经后置过滤剥离诊断性输出；RAG 实施两阶段硬匹配提升信噪比，并要求 LLM 照抄强引用 ID；病历模块实施防漂移自重试机制。若关键组件缺失判定为证据链断裂，触发降级提示交还医生裁量。
3. **数据归一化与非侵入原则**：应用域仅通过主索引建立路由映射逻辑聚合多源数据，严禁对医院现有异构源系统进行侵入性写入或直连表操作。
4. **计算转码与前端渲染分离原则**：前端规避引入重型 DICOM 解析库。所有医疗影像必须由推理域在后端预处理为 Web 标准格式，前端仅负责渲染与内存生命周期管控。
5. **异步与非阻塞原则**：应用域规避同步阻塞调用推理域。必须遵循异步 SSE 契约，且推理域必须将同步推理解耦至独立线程，配合异步事件循环实现断连步级终止，防止计算资源无效占用。
6. **契约驱动与服务隔离原则**：推理域与应用域协作严格遵循 API 契约定义的数据结构与流式事件格式，确保异构系统并行开发与集成。
