## 项目定位

DermaIntegrate 是一个**皮肤病多源数据集成与 AI 辅助诊断平台**。

### 核心链路

- 多模态数据录入（图像 / 病历文本 / 病理数据）
- Node/Express 应用域接收请求
- 转发到 FastAPI AI 推理域
- AI 侧异步生成 `task_id`
- 前端通过 SSE 订阅推理过程
- 分步展示中间结果与最终结构化报告

### 项目目标

- 作为简历项目，体现**AI 应用落地 + 前后端工程能力 + 联调能力**
- 展示对 **SSE、代理层、多模态 AI 联调、契约驱动开发** 的理解和实现能力
- 最终形成一个可演示、可讲述、可写进简历、可应对面试追问的完整项目

## 长期工作原则（每次会话都要遵守）

1. **先做最小 MVP，不追完整版**
    - 先把当前阶段的小闭环跑通
    - 不提前铺开后续阶段
    - 当前能跑、能测、能演示，比未来设计更重要

2. **核心链路优先，周边功能后置**
    - 先保 Node 代理层、上传、SSE、联调链路
    - 再考虑 FHIR、EMPI、正式前端、部署等扩展内容

3. **最小迭代原则**
    - 每次只做一个明确子任务
    - 每轮开始前先输出：
        - 为什么做它
        - 会改哪些文件
        - 最小实现方案
        - 风险
        - 验收标准
    - 用户确认后再动手

4. **按场景选择改动策略**
    - 新增功能、创建新文件、独立模块搭建  
      → 不必机械追求最小 diff，优先保证结构清晰和代码可读性
    - 修改核心链路、局部修 bug、联调修正  
      → 优先小改，降低风险和 token 消耗
    - 总原则：**改动要聚焦，不要无关扩散**

5. **每次迭代都必须测试**
    - 改完必须给出最小可执行验证方案
    - 不允许只写代码不验证
    - 优先给出：启动步骤、curl 命令、预期输出、失败时先查哪层

6. **不确定就直说，不要编造**
    - 说明书没写、代码没体现、当前没验证过的内容，要明确标为“待确认”
    - 不要把“推理得出”写成“已验证”
    - 具体字段、路径、接口行为、状态结论，必须有来源

7. **完成即更新文档**
    - 每完成一轮功能或修复，立刻更新 `docs/session-log.md`
    - 若阶段状态或任务清单变化，同步更新 `docs/dev-plan.md`
    - 若出现新的分歧、风险、联调问题，更新 `docs/known-issues.md`
    - 若产生新的学习点，更新 `docs/knowledge-base.md`
    - 若形成新的面试素材，更新 `docs/interview-qa.md`

8. **统一 UTF-8 编码**
    - 所有 Markdown 文档必须使用 UTF-8
    - 不允许乱码
    - 不允许出现 `????` 替代中文
    - 一旦发现编码异常，先修编码，再继续编辑内容

9. **每轮都要给学习型反馈**
    - 项目开发不只是产出代码，也要帮助用户真正理解
    - 每轮结束后默认补充：
        - 这轮你必须理解的核心概念
        - 你现在至少应该能解释的问题
        - 如果面试官问这个功能，你应该怎么回答
        - 这轮可以写进简历的点
        - 哪些点暂时可以不深究 / 下一轮再补

10. **控制 token 消耗**
    - 输出尽量简洁，优先结论、文件清单、执行步骤
    - 不做不必要的长篇分析
    - 默认只读取最小必要上下文
    - 不重复总结整个项目，除非用户明确要求

## 默认协作方式

当用户下达一个开发任务时，默认按以下顺序工作：

1. 先判断当前任务是不是自然的下一步
2. 如果不是，先指出更合理的顺序
3. 如果是，先给出：
    - 为什么做它
    - 准备修改哪些文件
    - 最小实现方案
    - 风险
    - 验收标准
4. 再开始动手
5. 完成后给出：
    - 修改了什么
    - 怎么测试
    - 学习型反馈
    - 下一步最自然做什么

除非用户明确要求，否则：
- 不要顺手扩展别的功能
- 不要一次性铺开太多方向
- 不要为了显得完整而过度设计
