## 第1章 引言

### 1.1 目的

本文档为 DermaIntegrate 平台的用例分析与模块设计说明书，供系统分析、开发、测试及运维人员使用。文档完整描述系统的参与者、业务用例、模块职责及非功能约束，作为后续详细设计和验收测试的基准。

### 1.2 范围

DermaIntegrate 平台由两个相对独立的技术域组成：

- **应用平台域**：负责用户交互、数据聚合、会话管理、对外接口代理。技术栈为 Vue 3 前端 + Node/Express 后端 + MySQL（derma_app / derma_his / derma_lis / derma_pacs 四库）。
- **智能推理域**：负责多模态 AI 推理，由协作方维护，对应用平台域表现为黑盒外部服务。技术栈为 FastAPI（Python）+ MySQL（derma_ai）+ Qdrant 向量库。

本文档覆盖应用平台域的全部功能需求，以及应用平台域与智能推理域之间的集成契约行为。智能推理域内部实现不在本文档范围内。

### 1.3 定义与缩略语

| 缩略语 / 术语 | 说明 |

## 第2章 整体描述

### 2.1 产品视角

DermaIntegrate 面向皮肤科临床场景，通过 EMPI 机制将分散在 HIS、LIS、PACS 三个异构系统中的患者数据统一关联，在临床视图中聚合呈现。当外部系统推送高价值模态数据（PACS 影像或 LIS 病理报告）时，平台自动触发多模态 AI 分析，并通过 SSE 流式协议将推理进度实时反馈给前端工作站。

AI 建议以内嵌侧栏形式呈现于临床视图旁，医生在查阅真实病历数据的同时可参考 AI 辅助建议。平台不替代医生的临床判断，AI 建议仅供参考，最终诊断责任归属于主治医生。

### 2.2 产品功能概览

| 功能域 | 说明 |
|---|---|
| 安全认证 | 基于 httpOnly Cookie 的登录/登出/会话保持 |
| 患者管理 | 患者信息增删改查，EMPI 跨系统身份映射 |
| 临床视图 | HIS 就诊、LIS 化验/病理、PACS 影像的多 Tab 聚合展示，AI 建议以内嵌侧栏形式呈现 |
| AI 自动触发 | 外部系统推送 PACS 影像或 LIS 病理报告时，系统自动发起多模态分析（防抖 30s，冲突打断旧任务） |
| AI 辅助建议 | SSE 接收实时推理进度，三个 agent 事件独立并行渲染，result 事件覆盖更新综合研判区 |
| 系统运维 | 账号管理、EMPI 映射治理、活跃 SSE 会话强制释放、AI 任务历史查阅 |
| 数据集成监控 | 外部系统数据推送沙箱、EMPI 运维面板 |

### 2.3 用户特征

| 角色 | 技术背景 | 使用频率 | 核心诉求 |
|---|---|---|---|
| 医生（Doctor） | 非技术，熟悉临床工作站操作 | 每日高频 | 快速定位患者、查看整合真实病历数据、在临床视图旁侧查阅 AI 辅助建议 |
| 应用域管理员（Admin） | 有 IT 背景，熟悉系统运维 | 低频，按需 | 账号管理、数据一致性维护、异常会话清理、AI 任务历史查阅 |
| 外部系统沙箱（External System） | 系统间集成，无人工交互 | 批量周期性推送 | 向平台写入 HIS/LIS/PACS 原始数据 |
| 智能推理域（AI Service） | 黑盒服务，协作方维护 | 由应用域自动触发 | 接收多模态数据包，通过 SSE 返回推理进度与结果 |

### 2.4 约束

1. **认证约束**：系统采用 httpOnly Cookie，前端不可通过 JavaScript 读取 token；EventSource 原生携带 Cookie，SSE 连接无需额外鉴权头。
2. **契约约束**：与智能推理域的交互严格遵循契约 v4.0.0；应用域不得假设推理域内部实现，不得调用推理域其他接口。
3. **SSE 渲染约束**：前端必须动态渲染 step 事件，严禁硬编码固定步骤顺序，收到什么渲染什么。三个 agent 事件（image_done / clinical_done / pathology_done）独立并行，顺序不预设。
4. **EMPI 约束**：外部系统接入通过向 empi_index 插行实现，DDL 零变更。
5. **数据库约束**：四库物理分离，跨库查询通过后端服务层聚合，不允许跨库 SQL JOIN。
6. **快照约束**：AI 推理 result 事件到达时必须持久化至 ai_tasks.result_snapshot，保证历史查阅与在线推理解耦。result_snapshot 主要供运维/管理员查阅，不作为医生主流程归档报告。
7. **触发约束**：AI 分析仅在 PACS 影像推送或 LIS 病理报告推送时自动触发；HIS 文字更新不单独触发，但下次触发时作为上下文带入；防抖窗口 30 秒。

### UC-01 安全登录与会话管理

| 字段 | 内容 |
|---|---|
| **用例 ID** | UC-01 |
| **用例名称** | 安全登录与会话管理 |
| **参与者** | 医生（主）、应用域管理员（主） |
| **前置条件** | 用户已进入登录页面（/login）；doctors 表中存在有效账号 |
| **触发事件** | 用户在 LoginView.vue 中输入凭据并提交登录表单 |
| **主成功流程** | 1. 用户在 /login 页面输入用户名与密码<br>2. 前端向后端发送 `POST /api/auth/login`（JSON body：username, password）<br>3. 后端查询 derma_app.doctors 表，bcrypt 校验密码<br>4. 验证通过，后端 Set-Cookie 响应头写入 httpOnly Cookie（含会话标识）<br>5. 前端收到 200 响应，跳转至 /patients（默认首页）<br>6. 后续请求浏览器自动携带 Cookie；后端通过 `GET /api/auth/me` 验证会话有效性 |
| **异常流程** | A1（凭据错误）：步骤 3 校验失败 → 返回 401 → 前端提示"用户名或密码错误"，停留登录页<br>A2（账号被禁用）：doctors.is_active = false → 返回 403 → 前端提示"账号已停用"<br>A3（网络超时）：前端请求超时 → 提示"服务不可用，请稍后重试" |
| **备选流程** | B1（已登录用户访问 /login）：后端 GET /api/auth/me 确认会话有效 → 前端直接重定向至 /patients |
| **后置条件** | 用户拥有有效会话 Cookie，可访问受保护路由 |
| **涉及接口** | `POST /api/auth/login`、`GET /api/auth/me`、`POST /api/auth/logout`（契约 v4.0.0） |

### UC-03 临床视图聚合

| 字段 | 内容 |
|---|---|
| **用例 ID** | UC-03 |
| **用例名称** | 临床视图聚合（HIS/LIS/PACS 多源数据展示） |
| **参与者** | 医生（主） |

| **备选流程** | 无 |
| **后置条件** | 医生在单一视图中看到患者完整跨系统临床数据；AI 辅助建议侧栏可同步查阅 |
| **涉及接口** | `GET /api/patients/:id/clinical-view`、`GET /api/empi/match` |

### UC-04 基于高价值模态推送自动触发 AI 分析

| 字段 | 内容 |
|---|---|
| **用例 ID** | UC-04 |
| **用例名称** | 基于高价值模态推送自动触发 AI 分析 |
| **参与者** | 系统内部（外部系统推送触发，无需医生手动操作） |
| **前置条件** | 外部系统已通过 UC-12 成功推送 PACS 影像记录或 LIS 病理报告；目标患者在 empi_index 中已有映射 |
| **触发事件** | WebhookRouter 检测到入库数据类型为 PACS 影像推送（写入 ext_pacs_records）或 LIS 病理报告推送（写入 lis_pathology_reports） |
| **主成功流程** | 1. WebhookRouter 接收外部推送并完成数据入库（见 UC-12）<br>2. WebhookRouter 判断推送类型：若为 PACS 影像或 LIS 病理报告，通知 DebounceManager<br>3. DebounceManager 检查同一 internal_patient_id 的防抖窗口（30 秒）：若窗口内已有待发任务，重置计时器；否则新建计时<br>4. 30 秒窗口到期，DebounceManager 向 TaskConflictResolver 提交发起请求<br>5. TaskConflictResolver 查询 derma_app.ai_tasks，检测同一 patient_id 是否存在 running 任务<br>6. 若存在 running 任务：TaskConflictResolver 向其发送中断信号（触发 UC-08 断连回收），将旧任务 status 更新为 interrupted，再发起新任务<br>7. 后端通过 empi_index 聚合该患者最新的 PACS image_url、HIS 就诊摘要（clinical_text）、LIS 病理数据（lab_json）<br>8. InferenceDispatcher 构造 multipart/form-data，向推理域 `POST /upload` 发送数据包<br>9. 推理域返回 202 Accepted，含 task_id<br>10. 后端在 ai_tasks 表插入新记录（task_id、patient_id、status=pending、pacs_record_id）<br>11. 系统记录触发日志：`{ event: "auto_trigger", trigger_type: "pacs"|"lis_pathology", patient_id, task_id }` |
| **异常流程** | A1（PACS 影像下载失败）：InferenceDispatcher 返回失败 → ai_tasks 记录 status=failed → 记录错误日志，等待下次推送触发重试<br>A2（推理域 /upload 不可用）：后端记录 status=failed，系统告警日志，不影响外部推送的 201 响应<br>A3（HIS/LIS 数据不完整）：仍发送可用字段，推理域返回降级报告（见 UC-06 A1）<br>A4（30 秒内同患者多次推送）：防抖窗口重置，合并为一次分析，避免重复触发 |
| **备选流程** | B1（HIS 文字更新推送）：WebhookRouter 判断类型为 HIS 就诊记录，不触发 AI 分析；数据正常入库，下次 PACS/LIS 推送时作为上下文带入 |
| **后置条件** | ai_tasks 表存在新任务记录（status=pending），task_id 已获取；前端可通过轮询或主动访问患者档案查阅 AI 建议进度 |
| **涉及接口** | `POST /api/tasks/from-pacs`（内部调用）、`POST /upload`（推理域）、`GET /api/tasks` |

|---|---|
| **用例 ID** | UC-05 |
| **用例名称** | SSE 流式 AI 推理进度实时展示 |
| **参与者** | 医生（主）、智能推理域（从） |
| **前置条件** | UC-04 已成功完成，task_id 已获取；前端已跳转至 /tasks/:taskId（TaskDetail.vue）或医生在临床视图 AI 建议侧栏中打开任务详情 |
| **触发事件** | TaskDetail.vue 组件挂载，前端创建 EventSource 连接至 `GET /api/tasks/:id/stream` |
| **主成功流程** | 1. 前端 TaskDetail.vue 挂载时创建 `EventSource`（浏览器自动携带 httpOnly Cookie）<br>2. 后端 routes/stream.js 鉴权后，向推理域建立 `GET /stream/{task_id}` 的上游 SSE 连接<br>3. 后端以 SSE 代理角色将推理域事件透明转发给前端<br>4. 推理域三个 agent 事件独立并行触发，顺序不预设，前端收到哪个渲染哪个：<br>　　- `event: step`，data.step = `image_done`（含 image_url、morphology、coverage、location）→ 前端追加图像分析卡片<br>　　- `event: step`，data.step = `clinical_done`（含 patient_info、lesion_clinical、lesion_symptoms）→ 前端追加临床信息卡片<br>　　- `event: step`，data.step = `pathology_done`（含 t_stage、treatment_recommendations、missing_data_warnings）→ 前端追加病理分析卡片<br>5. 推理域发送 `event: heartbeat` → 后端与前端均忽略，仅用于保活<br>6. 推理域发送 `event: result` → 触发 UC-06（综合研判覆盖更新）<br>7. 前端 EventSource 关闭，触发 UC-08 资源回收 |

| **涉及接口** | `GET /api/tasks/:id/stream`（后端 SSE 代理）、`GET /stream/{task_id}`（推理域上游） |

**SSE 事件流说明（动态触发，顺序不预设）：**

```
前端 EventSource              后端 SSE 代理               推理域 SSE 上游
      |                             |                            |
      |--- GET /api/tasks/:id/stream (Cookie) ->                 |
      |                             |--- GET /stream/{task_id} ->|
      |                             |                            |
      |     三个 agent 事件独立并行，任意顺序到达：               |
      |                             |<-- event:step(image_done) -|  ← 顺序不定
      |<-- event:step(image_done) --|                            |
      | [追加图像分析卡片]            |                            |
      |                             |<-- event:step(pathology_done) ← 可能先于 clinical
      |<-- event:step(pathology_done)                            |
      | [追加病理分析卡片]            |                            |
      |                             |<-- event:step(clinical_done)
      |<-- event:step(clinical_done)|                            |
      | [追加临床信息卡片]            |                            |
      |                             |<-- event:heartbeat --------|
      | [忽略]                       | [忽略]                      |
      |     result 事件在所有 agent 完成后到达（依赖前置）：        |
      |                             |<-- event:result -----------|
      |<-- event:result ------------|                            |
      | [触发 UC-06，覆盖更新综合研判区]                           |
      | EventSource.close()         | 关闭上游连接（触发 UC-08）   |
```

> 前端严禁硬编码步骤顺序。step 事件中 data.step 字段标识类型，前端按类型分区渲染，与到达顺序无关。

| **异常流程** | A1（EMPI 映射缺失）：EMPIEngine 找不到对应 internal_patient_id → 返回 422，`{ code: "EMPI_NOT_FOUND" }` → 外部系统记录失败，等待管理员在 UC-10 配置映射后重推<br>A2（数据格式不合法）：后端 Schema 校验失败 → 返回 400，`{ code: "INVALID_PAYLOAD", detail }` |
| **备选流程** | B1（无 EMPI 映射时自动注册）：若推送数据包含 id_card 且能唯一标识患者，系统自动在 empi_index 插入新行（需配置 auto_register 开关，默认关闭） |
| **后置条件** | 外部数据已写入对应分型表；下次调用 clinical-view 时可聚合展示；PACS/LIS 病理推送已触发 UC-04 自动分析流程 |
| **涉及接口** | `POST /api/mock/his_push`、`POST /api/mock/lis_push`、`POST /api/mock/pacs_push`（或通用 `POST /api/webhook/ingest`） |
| **触发说明** | 测试用例应构造符合约定的 JSON payload 发送至 Webhook 接口，不应直接操作数据库写入触发记录，以确保完整验证推送链路（含 EMPI 查询、分型路由、DebounceManager 通知）。 |

## 第5章 模块设计

### 5.1 模块总览

应用平台域划分为六大模块，各模块职责单一、边界清晰：

| 模块编号 | 模块名称 | 核心关注点 |
|---|---|---|
| M1 | 接入与安全网关层 | 认证、鉴权、路由分发 |
| M2 | 互操作性与数据集成层 | EMPI 映射、外部数据聚合、clinical-view |
| M3 | 诊疗会话状态管理层 | 任务状态机、result_snapshot 持久化 |
| M4 | 演进式推理编排层 | 多模态组装、上传推理域、智能触发策略 |
| M5 | SSE 代理与生命周期管理层 | 流式透传、断连感知、资源回收 |
| M6 | 业务治理与审计层 | 账号管理、EMPI 治理、会话治理、审计日志 |

---

### M1 接入与安全网关层

**职责**：统一认证、RBAC 鉴权与路由分发。所有请求进入业务逻辑前必须通过本层。

| 核心组件 | 职责 |
|---|---|
| AuthMiddleware | 校验 httpOnly Cookie，提取 user_id 和 role（doctor/admin），写入 req.user |
| RBACGuard | 路由级拦截：/api/doctor/* 仅限医生，/api/admin/* 仅限管理员，未授权返回 403 |
| WebhookRouter | 接收外部系统推送，校验 API 密钥，路由至对应数据写入服务，并在 PACS/LIS 病理推送后通知 DebounceManager |

**关键接口**：
- `POST /api/auth/login` — 凭据验证，Set-Cookie
- `POST /api/auth/logout` — 清除 Cookie
- `GET /api/auth/me` — 返回当前会话用户信息

---

### M2 互操作性与数据集成层

**职责**：解决数据孤岛，完成跨系统患者身份映射与多源数据聚合。

| 核心组件 | 职责 |
|---|---|
| EMPIEngine | 执行 `(source_system, external_id) → internal_patient_id` 的实时映射查询；支持反向查询（internal_patient_id → 各系统 external_id） |
| ClinicalViewAggregator | 并发查询四库（his/lis/pacs），将 HIS 就诊 + LIS 化验/病理 + PACS 影像合并为统一响应结构 |
| FHIRAdapter | 将外部非标准 JSON（HIS/LIS 推送）转为内部 FHIR R4 风格资源（Encounter、DiagnosticReport），便于统一处理 |

**关键接口**：
- `GET /api/patients/:id/clinical-view` — 四源聚合
- `GET /api/empi/match` — EMPI 查询
- `GET /api/empi/sources` — 外部数据源列表统计

---

### M3 诊疗会话状态管理层

**职责**：维护 AI 任务生命周期状态，保障历史快照与在线推理解耦。

| 核心组件 | 职责 |
|---|---|
| TaskStateManager | 管理 ai_tasks 表的状态流转：pending → running → completed/failed/interrupted |
| SnapshotPersistor | 收到推理域 result 事件时，将完整 result JSON 序列化写入 ai_tasks.result_snapshot；result_snapshot 持久化后主要供运维/管理员查阅，不依赖推理域在线 |

**状态机**：

```
[pending] → [running] → [completed]
               ↓            ↓
           [failed]    [interrupted]
```

**关键接口**：
- `GET /api/tasks` — 任务列表（支持 pending 静默轮询）
- `GET /api/tasks/:id` — 任务详情（含 pacs_image_url）
- `GET /api/tasks/:id/result` — 历史快照读取

---

### M4 演进式推理编排层

**职责**：多模态数据组装与推理域调用，实现基于推送事件的智能触发策略。

| 核心组件 | 职责 |
|---|---|
| InferenceDispatcher | 聚合 file + clinical_text + lab_json，构造 multipart/form-data，调用推理域 `POST /upload`，获取 task_id |
| DebounceManager | 接收 WebhookRouter 的触发通知；对同一 patient_id 维护 30 秒防抖窗口：窗口内重复推送重置计时，窗口到期向 TaskConflictResolver 提交发起请求 |
| TaskConflictResolver | 新任务提交时，检测同一 patient_id 是否存在 running 任务；若冲突，向旧任务发送中断信号（触发 UC-08），将旧任务标记为 interrupted 后发起新任务 |
| MultimodalPreprocessor | 从 PACS 下载图像文件流；将 HIS 就诊数据序列化为 clinical_text（包含本次触发前的最新 HIS 记录）；将 lis_pathology_reports 数据格式化为 lab_json（对齐 TCGA-SKCM/AJCC 8th 字段） |

**智能触发策略**：
- 仅 PACS 影像推送或 LIS 病理报告推送触发自动分析
- HIS 文字更新不单独触发，但下次触发时作为最新上下文带入 clinical_text
- 防抖窗口 30 秒，避免短时间内多次推送重复发起任务

**关键接口**：
- `POST /api/tasks/from-pacs` — 内部调用，由 DebounceManager 发起多模态分析
- `POST /api/tasks/upload` — 手动上传文件发起分析（备用路径）

---

### M5 SSE 代理与生命周期管理层

**职责**：流式数据透传与基于契约的生命周期管理，防止推理域资源被无效占用。

| 核心组件 | 职责 |
|---|---|
| SSEProxy | 建立与推理域 `GET /stream/{task_id}` 的上游连接，以流式方式透传 step/result/error 事件给前端；heartbeat 事件在应用域层过滤，不透传给前端 |
| ConnectionMonitor | 注册前端 response 对象（内存 Map，key=task_id）；监听 `response.on('close')` 事件，感知前端断连；触发 ResourceReclaimer |
| ResourceReclaimer | 接收断连通知，调用对应上游 EventSource.close()，关闭应用域→推理域连接；幂等设计，重复调用安全 |

**SSE 代理超时配置**：Node/Express 代理 SSE 时读取超时必须 > 5 分钟，防止长连接被提前切断。

**关键接口**：
- `GET /api/tasks/:id/stream` — 后端 SSE 代理，对外暴露给前端
- `/ai-static/*` — 推理域静态资源代理（热力图等）

### M6 业务治理与审计层

**职责**：为管理员提供应用域范围内的业务运维与安全审计能力。

| 核心组件 | 职责 |
|---|---|
| UserManagementService | 医生账号 CRUD、密码重置（bcrypt 加密）、状态冻结与软删除 |
| EMPIMgmtService | empi_index 的增删改查界面服务，支持冲突扫描，供管理员手动纠正错误 ID 绑定 |
| SessionGovernor | 从 ConnectionMonitor 内存注册表读取所有活跃医生接诊会话；管理员点击"强制释放"时调用 ConnectionMonitor 提供的方法主动关闭指定前端 HTTP response，安全触发断连回收链路 |

**关键接口**：
- `GET /api/admin/users`、`POST/PATCH/DELETE /api/admin/users/:id`

## 附录A：EMPI 实现方案对比说明

### A.1 参考方案（PDF 文档中的方案）

PDF《一、EMPI（主索引映射）的模拟与实现》描述了如下表结构：

```sql
CREATE TABLE empi_mapping (
  id INT AUTO_INCREMENT PRIMARY KEY,
  unified_patient_id VARCHAR(64) UNIQUE NOT NULL,  -- 内部唯一 ID
  id_card VARCHAR(32) UNIQUE NOT NULL,             -- 身份证号（联邦键）
  his_patient_id VARCHAR(64),                      -- HIS 外部 ID
  pacs_patient_id VARCHAR(64),                     -- PACS 外部 ID
  lis_patient_id VARCHAR(64),                      -- LIS 外部 ID
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

查询路径：`his_patient_id` → `EMPIEngine` → `unified_patient_id`，再以 `unified_patient_id` 拉取多源数据。

**该方案的问题：**

1. **扩展需改表**：每接入一个新外部系统（如 EMR、RIS），必须 `ALTER TABLE ADD COLUMN`，即 DDL 变更，存在锁表风险。
2. **NULL 列蔓延**：对于没有 PACS 数据的患者，`pacs_patient_id` 一列全为 NULL，浪费存储且语义模糊。
3. **索引低效**：三个外部 ID 列各需单独索引，随系统增多索引数量线性增长。

### A.2 当前系统方案

当前系统 `empi_index` 表采用行键设计：

```sql
CREATE TABLE empi_index (
  id INT AUTO_INCREMENT PRIMARY KEY,
  source_system VARCHAR(32) NOT NULL,       -- 来源系统标识：his/lis/pacs
  external_id   VARCHAR(64) NOT NULL,       -- 该系统内的外部 ID
  internal_patient_id VARCHAR(64) NOT NULL, -- 内部患者 ID
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_source_external (source_system, external_id)
);
```

查询路径：`(source_system='his', external_id='H8392')` → `internal_patient_id`。

**当前方案的优势：**

| 维度 | PDF 方案 | 当前方案 |
|---|---|---|
| 接入新外部系统 | DDL 改表（ALTER TABLE） | 仅插新行，零 schema 变更 |
| NULL 列问题 | 存在（无数据的列为 NULL） | 不存在（无数据则无行） |
| 索引设计 | 每个外部 ID 列单独索引 | 一个复合唯一键覆盖全部系统 |
| 扩展性 | 受限，每个新系统改一次表 | 无限扩展，行数增长而非列数增长 |
| 语义清晰度 | source 信息隐含在列名中 | source_system 字段显式存储 |

### A.3 结论

**当前系统的 `(source_system, external_id)` 复合行键方案更优**，是标准的 EMPI 行键设计模式。接入新外部系统只需插行，DDL 零变更，扩展性更强，NULL 列问题完全消除。后续如需接入 EMR（电子病历系统）或 RIS（放射信息系统），只需在 empi_index 插入 `source_system='emr'` 的行即可，无需修改表结构。
