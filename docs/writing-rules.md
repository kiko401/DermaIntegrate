# writing-rules.md — 项目文档写作规则

> 本规则适用于本项目所有文档。AI 助手和人工协作者均须遵守。
> 建立于 2026-06-08，来源：首轮文档纠偏。

---

## 一、事实状态三档

项目文档中涉及项目状态、模块完成度、接口对齐、联调结论、问题状态时，每条结论必须属于以下三档之一，并注明动作来源：

| 标记 | 含义 | 要求 |
| :--- | :--- | :--- |
| ✅ 已验证 | 亲自执行了某个动作，结果可重现 | 必须注明动作来源（见下） |
| 🔶 推理得出 | 基于已验证事实做出的合理推断 | 必须注明推断依据 |
| ⬜ 待确认 | 当前看不到、没跑过、没验证过 | 不能用已验证或推理的语气表述 |

### 合法的动作来源（"已验证"必须能对应其中一项）

- **读源码**：用 Read 工具读取了该文件，读到了具体内容
- **grep**：用 Grep 工具在代码中搜索到了该符号/字段
- **git show**：通过 `git show <branch>:<path>` 读取了指定分支的文件内容
- **curl / 联调**：实际发起了 HTTP 请求，收到了响应
- **用户明确提供**：用户在对话中直接给出了该事实（标注为"用户口头确认"，仍需区分是否有代码/部署验证）

## 二、写文档前的自查清单

在写下任何关于项目状态的句子之前，问自己：

1. 这个文件我实际读过了吗？  
   → 没读过：不能写成"已存在并实现了 X 功能"
2. 这个接口我 curl / 联调过了吗？  
   → 没跑过：不能写成"接口正常"或"已验证"
3. 这段描述来自 README / 契约 / 规划文档？  
   → 必须标注"文档描述"或"目标架构"，不能写成当前完成态
4. 只有用户对话确认，没有代码/部署/联调验证？  
   → 写"用户口头确认，待代码/部署/联调验证"
5. 这条"已解决"的依据是什么？  
   → 必须注明：代码验证 / 部署验证 / 联调验证 / 仅用户口头确认

## 三、禁止写法

| 禁止 | 原因 | 替换写法 |
| :--- | :--- | :--- |
| 把 README 功能描述写成已完成实现 | README 是规划文档，不是验收报告 | 标注"来自 README 描述，推理得出；实际实现待读源码确认" |
| 把 docker-compose 端口写成云端运行状态 | 配置文件 ≠ 运行状态 | 标注"来自 docker-compose.yml 配置；云端实际运行状态待 curl 确认" |
| "已解决（待部署验证）" | 自相矛盾 | 降级为"用户口头确认，待部署/联调验证" |
| 没读过的文件写成"已存在并完成" | 目录树存在 ≠ 内容已读 ≠ 功能已实现 | 拆成"文件已存在（来自目录树）"和"功能描述（来自 README）" |
| "可联调"但没有 curl 过 | 地址在文档里 ≠ 服务当前在线 | 标注"来自契约文档；实际在线状态待 curl 确认" |

---

## 四、"已解决"的升级条件

一个问题只有满足以下条件之一，才能标记为 ✅ 已解决：

- **代码验证**：通过 Read / git show 读到了修复后的代码
- **部署验证**：curl 或运行日志确认修复已在目标环境生效
- **联调验证**：端到端联调中确认该问题不再出现

仅有用户对话确认 → 保持 🟡，标注"用户口头确认，待 [代码/部署/联调] 验证"。
