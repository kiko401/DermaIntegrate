# 系统设计概览（对齐版）

**面向**：AI 域协作方（智能推理域设计者）
**日期**：2026-06-18
**对应契约版本**：v4.0.0

---

## 一、系统定位

DermaIntegrate 是皮肤病多源数据集成与 AI 辅助诊断平台，分为两域：**应用平台域**（Vue3 + Node/Express + MySQL 四库）和**智能推理域**（FastAPI，黑盒，协作方维护）。医生打开患者档案，看到真实病历数据（HIS/LIS/PACS），旁侧内嵌 AI 辅助建议；AI 建议不替代诊断，只辅助参考，不归档为正式诊断文书。

---

## 二、参与者

| 参与者 | 说明 |
|---|---|
| 医生 | 查看患者档案和临床数据，旁侧查阅 AI 辅助建议 |
| 管理员 | 账号管理、EMPI 治理、会话监控、历史 AI 快照查阅 |
| 外部系统沙箱 | 模拟 HIS/LIS/PACS 向平台推送原始数据；PACS 影像或 LIS 病理报告推送自动触发 AI 分析 |
| 智能推理域 | 接收应用域的多模态数据包，通过 SSE 返回推理进度与结果 |

## 四、AI 分析触发机制

**触发条件（仅以下两种）：**
- PACS 影像记录推送（写入 `ext_pacs_records`）
- LIS 病理报告推送（写入 `lis_pathology_reports`）

**不触发的情况：**
- HIS 就诊记录文字更新不单独触发 AI 分析
- HIS 数据仍会入库，下次触发时作为最新 `clinical_text` 上下文带入

**防抖策略：**
- 同一患者在 30 秒窗口内的多次推送合并为一次分析
- 窗口内重复推送重置计时器，窗口到期才发起任务

**冲突处理：**
- 同一患者存在 `running` 状态的任务时，TaskConflictResolver 向旧任务发送中断信号
- 旧任务标记为 `interrupted`，发起新任务
- 不静默忽略，确保最新推送数据对应最新分析

## 五、SSE 事件流设计

> 这是对智能推理域最重要的对齐内容。

### 5.1 事件类型与处理方式

| 事件 | data.step | 应用域处理 | 前端处理 |
|---|---|---|---|
| `event: step` | `image_done` | 透传给前端 | 渲染图像分析卡片（含 image_url、morphology、coverage、location） |
| `event: step` | `clinical_done` | 透传给前端 | 渲染临床信息卡片（含 patient_info、lesion_clinical、lesion_symptoms） |
| `event: step` | `pathology_done` | 透传给前端 | 渲染病理分析卡片（含 t_stage、treatment_recommendations、missing_data_warnings） |
| `event: heartbeat` | — | 应用域过滤，不透传 | 不接收 |
| `event: result` | — | 透传 + 持久化 result_snapshot | 覆盖更新综合研判区 |
| `event: error` | — | 透传给前端 | 展示错误提示，关闭 EventSource |

### 5.2 三个 agent 事件的关键约定

- `image_done` / `clinical_done` / `pathology_done` **三个事件独立并行触发，顺序不预设**
- 应用域按收到顺序透传，不重排、不等齐
- 前端按 `data.step` 字段值路由到对应渲染区，**严禁硬编码步骤顺序**
- `result` 事件（综合研判）依赖前置 agent 完成，会晚于三个 step 事件到达

### 5.3 断连回收链路

```
前端 EventSource.close()
        ↓
后端 response.on('close') 触发
        ↓
ConnectionMonitor 通知 ResourceReclaimer
        ↓
ResourceReclaimer 调用 upstreamSSE.close()
        ↓
推理域感知上游 HTTP 连接关闭
        ↓
推理域销毁 Multi-Agent 管线，释放资源
```

- 整条链路由前端断连自动触发，无需推理域主动轮询
- 应用域 UC-08 幂等设计，重复触发安全

## 六、模块划分

| 模块 | 名称 | 核心职责 |
|---|---|---|
| M1 | 接入与安全网关层 | httpOnly Cookie 认证、RBAC 鉴权（doctor/admin）、Webhook 入口 |
| M2 | 互操作性与数据集成层 | EMPI 行键映射、四库并发聚合、clinical-view 接口 |
| M3 | 诊疗会话状态管理层 | ai_tasks 状态机、result_snapshot 持久化（供运维查阅） |
| M4 | 演进式推理编排层 | DebounceManager（防抖 30s）、TaskConflictResolver（冲突打断）、多模态数据组装 |
| M5 | SSE 代理与生命周期管理层 | 透传 step/result/error、过滤 heartbeat、断连感知与资源回收 |
| M6 | 业务治理与审计层 | 账号管理、EMPI 治理、会话强制释放、审计日志 |

## 七、架构约束

- 前端不直连推理域，所有 AI 相关请求必须经应用域后端代理
- 应用域与推理域的跨域交互仅靠两个接口：`POST /upload`（上传数据包）+ `GET /stream/{task_id}`（SSE 事件流）
- MySQL 四库物理分离（derma_app / derma_his / derma_lis / derma_pacs），禁止跨库 JOIN，聚合由后端服务层完成
- EMPI 采用 `(source_system, external_id)` 复合行键设计，接入新外部系统只需插行，DDL 零变更
- 认证采用 httpOnly Cookie，SSE 连接由浏览器自动携带 Cookie，无需额外鉴权头
- `result_snapshot` 在 result 事件到达时持久化，历史快照查阅不依赖推理域在线
